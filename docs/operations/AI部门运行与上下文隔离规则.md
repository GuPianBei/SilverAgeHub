# AI 部门运行与上下文隔离规则

版本：2026-06-20 v1

## 一、窗口划分

每个窗口只能承担一个角色：

- 总控/产品窗口：需求、优先级、任务拆解、验收、跨部门决策
- 研发窗口：代码、云函数、数据库、联调
- UI 窗口：页面视觉、交互和适老化检查

禁止在研发窗口临时讨论产品扩展，也禁止在 UI 窗口直接重构云函数。

## 二、单窗口任务限制

每轮只允许：

- 一个明确目标
- 一个主要模块
- 一份验收清单
- 一次阶段性交付

任务预计超过 60 至 90 分钟，或涉及三个以上模块时，必须拆轮次或开新窗口。

## 三、单一事实源

窗口不能以旧聊天记忆作为最终依据，必须读取：

1. `README.md`
2. `docs/handoff/CURRENT_HANDOFF.md`
3. 当前部门指令
4. 当前任务对应的架构或验收文档

优先级：

```text
最新交接单
> 稳定架构文档
> MVP 验收清单
> 部门指令
> 历史自测记录
> 聊天记忆
```

## 四、文件所有权

- 总控/产品：`README.md`、`docs/product`、`docs/handoff`、验收和规则文档
- 研发：`miniprogram`、`cloudfunctions`、`database`
- UI：页面 WXML/WXSS 和 UI 规范；修改 JS 前必须通知研发

总控可以做紧急架构修复，但修复后必须更新交接单。

## 五、新窗口启动协议

新窗口开始时必须：

1. 扫描当前工作区，不假设文件未变化。
2. 阅读交接单和本部门指令。
3. 用不超过 10 行复述当前状态。
4. 明确本轮唯一目标和不做事项。
5. 先检查再修改。

## 六、交付协议

每轮结束必须更新交接单，并输出：

- 完成内容
- 修改文件
- 验证结果
- 未完成事项
- 风险
- 下一窗口第一步

不能只回复“已完成”，也不能把未验证内容写成已完成。

## 七、防污染规则

- 不把旧方案和新方案混在同一执行指令中。
- 历史记录可以保留，但必须标注“已废弃”或“仅作历史参考”。
- 新窗口不粘贴完整旧聊天，只提供交接文件路径。
- 同一问题连续失败两次时停止试错，回到架构和日志定位。
- AI 提出额外功能时默认不做，除非总控批准。
- 对字段、接口、权限和部署流程的修改必须同步文档。

## 八、窗口结束条件

出现任一情况就应结束当前窗口并交接：

- 上下文开始反复引用旧方案
- 同一文件被多个角色反复修改
- 当前任务范围已经明显扩大
- 已完成阶段目标
- 需要切换部门角色
- 回答速度明显下降且历史信息过多
