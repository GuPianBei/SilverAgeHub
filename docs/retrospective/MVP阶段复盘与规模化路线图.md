# 银发纪统一规模化路线图

版本：2026-06-20 v2

本文件是银发纪 MVP 之后唯一的产品、研发、运营和上架主路线图。

## 一、当前基线

银发纪 MVP 已完成：

```text
教程学习
→ 课程查看
→ 用户报名
→ 我的预约
→ 管理端维护
→ 预约确认
→ 用户端同步状态
```

已验证：

- 用户端主包与管理端分包隔离
- 用户端统一使用 `userApi`
- 管理端统一使用 `adminApi`
- `guide`、`course`、`order`、`admin_user` 已真实读写
- 管理员 OpenID 权限校验通过
- 教程、课程 CRUD 和预约状态更新通过
- 用户端适老化 UI 已达到 MVP 要求

## 二、阶段复盘

### 做得好的地方

- 产品范围始终围绕教程、课程、预约和管理
- 不只完成页面，真实跑通了云端业务闭环
- 发现四函数并存问题后，及时收敛为双 API 架构
- 建立产品、研发、UI 分工和 AI 窗口交接规则
- 通过版本探针、联调记录和验收清单减少假完成

### 主要问题

- 早期架构边做边定，出现两套云函数方案
- 本地代码与云端部署状态一度不同步
- 多个 AI 窗口引用旧方案，产生上下文污染
- 部分文档更新晚于代码
- 当前系统只验证了正常报名流程，尚未覆盖高并发、重复提交和运营异常

### 固化规则

- 同一能力只能有一个正式入口
- 架构变化先形成决策，再进入编码
- 云函数部署后先验证版本，再测试业务
- 新窗口只读取最新交接单和本路线图
- 每轮只处理一个主要目标
- 文档、代码和云端状态必须同时更新

## 三、总目标

下一阶段不是盲目加功能，而是把 MVP 升级为可真实运营、可审核上架、可稳定扩展的课程报名应用。

```text
MVP 基线
→ v0.2 内部测试版
→ v0.5 社区测试版
→ v0.9 审核候选版
→ v1.0 正式版
→ 多社区规模化
```

## 四、扩展原则

### 允许立即增加

- 报名幂等
- 人数并发控制
- 完整状态机
- 分页、搜索和筛选
- 报名名单导出
- 权限、操作日志和手机号脱敏
- 环境隔离、备份、监控和回滚
- 隐私、客服和运营流程

这些属于生产化能力，不属于范围膨胀。

### 暂不增加

- 社交系统
- 实时聊天
- AI 自动生成课程
- 数字人
- 复杂推荐算法
- 具身智能硬件接入
- 独立 Web 管理后台
- 未确认收费模式前的正式支付系统

### 规模原则

- 一次只扩大一个维度：用户、社区、内容、运营人员或功能
- 先扩大真实使用，再增加大型业务模块
- 新字段先兼容旧数据，再逐步迁移
- 新功能优先使用开关，先向测试账号或单个社区开放

## 五、v0.2 内部测试版

目标：课程报名逻辑在重复提交、并发和异常场景下保持正确。

### P0 业务能力

#### 报名幂等

同一用户对同一课程只能存在一条有效报名。

幂等键建议：

```text
openid + courseId
```

前端防重复点击只负责体验，服务端必须保证重复请求不会生成多条记录。

#### 人数并发控制

报名必须在云函数事务或原子操作内完成：

1. 读取课程状态
2. 检查重复报名
3. 检查剩余名额
4. 创建预约
5. 更新已占名额
6. 任一步失败则回滚

禁止使用“先查人数、前端判断、再创建预约”的方案。

#### 课程状态机

- `draft`：草稿
- `published`：可报名
- `full`：已满员
- `closed`：已截止
- `finished`：已结束
- `cancelled`：已取消

#### 报名状态机

- `pending`：未确认
- `confirmed`：已确认
- `completed`：已完成
- `cancelled`：已取消

迁移期间兼容现有中文状态，不直接破坏旧记录。

#### 课程快照

预约记录保存：

- 课程名称
- 上课时间
- 上课地点

课程修改或删除后，历史预约仍然可读。

#### 取消与名额释放

- 用户取消后释放名额
- 管理员取消需记录原因和操作人
- 课程取消或截止后禁止新增报名
- 有历史预约的课程不得直接物理删除

### P0 管理能力

- 预约和课程列表分页
- 关键词搜索
- 按课程、状态和时间筛选
- 显示总人数、已确认人数和剩余名额
- CSV 报名名单导出
- 手机号默认脱敏
- 导出完整手机号需要更高权限
- 记录导出人、时间和筛选条件

CSV 导出统一由 `adminApi` 执行，并通过临时云存储地址下载。

### P0 内部控制

- `admin`：管理员和权限配置
- `operator`：内容、课程和预约运营
- `viewer`：只读查看

需要记录：

- 新增、编辑、删除或取消课程
- 修改报名状态
- 导出名单
- 修改管理员

### v0.2 验收

- 多人同时抢最后一个名额不会超额
- 同一用户连续提交只产生一条报名
- 网络超时重试不会重复占名额
- 课程截止瞬间提交结果一致
- 取消报名后名额正确释放
- 管理端可以分页筛选和导出
- 高风险操作有日志

## 六、v0.5 社区测试版

目标：1 个社区、10 至 30 名真实用户和助教连续使用。

### 环境与发布

- 开发、测试和生产环境分离
- 固定依赖版本
- 自动化静态检查和核心云函数测试
- 发布检查单
- 数据备份和至少一次恢复演练
- 版本号、变更记录和回滚说明

### 隐私与数据

- 明确姓名、手机号、OpenID 的收集目的
- 页面手机号默认脱敏
- 日志不记录完整手机号
- 导出文件短期有效并定期清理
- 建立用户数据更正和删除流程
- 区分测试数据和正式数据

### 真实设备测试

- Android 和 iPhone
- 不同屏幕尺寸
- 大字体模式
- 较低微信基础库版本
- 弱网、断网和网络恢复
- 老年用户真实操作

### 运营闭环

明确：

- 谁每天处理新预约
- 多久内完成确认
- 谁联系用户
- 谁处理取消、改期和投诉
- 谁维护教程和课程
- 谁每周复盘数据

建议形成服务标准：

- 新预约处理时限
- 课程变化通知时限
- 用户反馈入口
- 严重故障联系人

### 核心指标

- 教程详情打开人数
- 课程详情到报名转化率
- 报名成功率
- 重复报名拦截次数
- 满员率
- 取消率
- 预约确认率
- 实际到课率
- 二次使用率
- 助教处理时长

## 七、v0.9 审核候选版

目标：具备微信审核和正式发布条件。

必须完成：

- 关闭普通用户管理入口
- 清理测试数据和过期导出文件
- 确认小程序、公众号和云环境主体归属
- 至少设置两名可信管理员
- 配置正式名称、图标、介绍和服务类目
- 完成隐私保护指引
- 准备审核账号、审核路径和说明
- 完成全链路、弱网、并发和真机回归
- 生成体验版并邀请指定人员验证
- 确认生产数据备份和回滚方案

## 八、支付系统决策

### 当前结论

当前正式决定：银发纪现阶段所有课程按照免费报名设计，不接入微信支付，不收取课程费用或预约保证金。

研发阶段不得：

- 创建支付订单集合
- 接入微信支付或退款接口
- 申请或配置商户号
- 在用户页面展示价格、付款或退款入口
- 为尚未成立的收费模式预埋复杂支付状态机

社区或政府补贴如需记录，先作为运营信息处理，不向用户发起支付。

### 适合接入的时间

只有未来同时满足以下条件，才重新立项评估支付：

- 已有真实收费课程
- 收款主体、商户号和结算账户明确
- 课程价格、退款和取消规则明确
- 已完成报名幂等、人数并发和状态机
- 有人负责财务、退款和用户投诉

届时必须作为独立项目评审，不直接在现有报名逻辑上临时增加付款按钮。

## 九、v1.0 正式版

目标：通过审核、关联公众号并开始真实运营。

### 发布

- 提交微信审核
- 审核通过后发布
- 上线后每日检查错误、报名和成本
- 严重故障可快速关闭报名或回退版本

### 公众号联动

- 公众号负责内容传播、活动通知、课程预告和用户召回
- 小程序负责教程学习、课程详情、报名和预约状态

推荐路径：

```text
公众号文章或菜单
→ 指定教程或课程页面
→ 小程序完成报名
→ 我的预约查看状态
```

关联方式、账号限制、通知能力和审核规则，以微信公众平台当日后台为准。

### 通知

规划：

- 报名成功确认
- 课程确认、改期或取消通知
- 上课前提醒
- 通知失败后的人工补救

不能假设关注公众号后即可无限发送消息。

## 十、100 至 1000 用户：多社区复制

进入条件：

- 单社区连续稳定运行至少四周
- 用户有持续学习和报名行为
- 助教可以独立完成运营
- 报名高峰、导出和数据恢复通过验证

新增能力：

- `organization/community` 组织维度
- 多社区数据隔离
- 社区管理员权限
- 候补名单
- 批量通知
- 内容草稿、审核、发布、下架
- 更完整的数据看板
- 成本和容量告警

此时再评估小程序管理分包是否成为效率瓶颈。

## 十一、1000 用户以上：区域化运营

进入条件：

- 多个社区形成稳定运营流程
- 内容和课程供给持续
- 小程序管理端无法满足批量运营

建设：

- 独立 Web 管理后台
- 更细的 RBAC 和数据权限
- 批量导入、导出和审核
- 监控、告警、成本分析
- 灰度发布和功能开关
- 媒体压缩与 CDN 策略
- 数据保留、归档和删除机制
- 客服工单和事件响应流程

## 十二、企业治理清单

### 资产

- 小程序、公众号、云环境和代码仓库归属明确
- 管理员、认证邮箱和手机号有备份
- 人员离职时及时回收权限
- 图片和视频拥有使用权

### 内容

- 防诈骗、健康等内容有可信来源
- 教程和课程经过审核后发布
- 过期内容定期复查和下架

### 故障

- 用户可见错误编号
- 云函数日志和异常告警
- 数据备份和恢复演练
- 发布回滚说明
- 数据误删、隐私问题和大量错误报名的应急流程

### 成本

监控：

- 云函数调用次数和耗时
- 数据库读写量
- 图片、视频存储和下载流量
- 日志量
- 导出文件数量

扩容前验证报名高峰、管理大列表、批量导出和媒体加载。

### 当前 P0：v0.2

2. 人数并发控制

5. 管理端分页筛选
6. CSV 导出
7. 手机号脱敏
8. 操作日志

### 当前 P1：v0.5

1. 角色权限
2. 环境隔离
3. 隐私和数据删除
4. 备份与恢复
5. 弱网、真机和并发测试
6. 用户反馈和运营责任

### 当前 P2：v0.9 至 v1.0

1. 审核和发布准备
2. 公众号关联
3. 通知与召回
4. 运营指标

### 暂缓

1. 独立 Web 管理后台
2. 社交和聊天
3. AI 内容和复杂推荐
4. 数字人和具身智能
